# [GHOST]   tag appears at the end of the summary line:
#           - one or more 'orphan' events were encountered which triggered the creations
#             of a 'ghost' request.  For example first event for an io context is an 'I'
#             results in a 'ghost' with synthesized 'Q' event.  Ghost events do not affect
#             inProgress, InFlight or any other io statistics -- they are ghosts.
# [Barrier] tag appears at the end of the summary line:
#           - a sequence of A,Q,G,I,C for sector:length 0:0 was detected. There are several
#             unusual attributes associated with this sequence including only the A event
#             has a specified sector:length (0 + 0) in the event -- all others have neither.
#             this appears to be uses as some type of io barrier.  there is no 'D'ispatch
#             as there is no data movement due to length=0.  There is special code to detect
#             this particular sequence and it gets flagged as Special Case.1.
# [BARRIER] tag appears at the end of the summary line:
#           - a non-read io with a sector:length value of 0:0 is detected, for 
#             example 'FWS 0 + 0' is a Flush/Write/Sync io of zero length to sector 0.
#             such io will not have a 'D'ispatch event and are considered a generic 
#             io barrier and are so tagged.  The difference between this and [Barrier] above
#             is that the above is a more specific 4.18 event stream type that requires
#             additional 'special' handling with the code whereas a generic io barrier is
#             otherwise the same as any normal io (with the exception of the missing
#             'D'ispatch event because the io has a zero-length size.

# . NVMe : NVMe device detected within blktrace capture log file.
#          NVMe devices have ioscheduler <none>, which means it does not have 'I'nsert events.  Within some
#          kernel versions there are no 'D'ispatch events, as a 'Q'ueue implies a 'D'ispatch event.
#          For NVMe devices the 'I' and 'D' events are set to the 'Q'ueue time if they are found to be 0.0.
#         This will result in Q2I (and in some cases Q2D) delta times to always be zero.  In those cases where
#           there was no 'D' event, the D2C(dt) will be the same as Q2C times as a result.

# === Time Descriptions ================================================================= 
#  (*)  Q2Q — time between requests sent to the block layer
#  (*)  Q2G — how long it takes from the time a block I/O is queued to the time it gets a request allocated for it
#  (*)  G2I — how long it takes from the time a request is allocated to the time it is Inserted into the device's queue
#       Q2I - how long it takes from the time a block I/O is assigned to a new request and inserted into the scheduler queue[1]
#  (*)  Q2M — how long it takes from the time a block I/O is queued to the time it gets merged with an existing request
#       I2D — how long it takes from the time a request is inserted into the device's queue to the time it is actually issued to the device
#  (*)  M2D — how long it takes from the time a block I/O is merged with an exiting request until the request is issued to the device
#       D2D - how long it takes from the first time I/O is issued to the device to the last time I/O was re-issued to the device [2]
#       D2C — service time of the request by the device, this time is dominated by storage latency[3]
#       Q2C — total time spent in the block layer for a request
# 
#  (*)  indicates this delta time value is not output within the merged one-line per request output that follows
#       but is available in other output files from standard blktrace package utilities. Typically these times are included
#       within combined times in other columns, for example Q2I = Q2G + Q2I. Nominally the main N2N times of interest are Q2D
#       D2C, and Q2C.  Only when 'S'leep events or 'R'equeue events occur is breaking down Q2D time for analysis of interest.
#  [1]  Q2I is the combination of Q2G + G2I
#  [2]  D2D will be non-zero only if there are 'R'equeue events present for the given I/O
#  [3]  service time includes the short code path within the driver, time spent within the HBA, transport time to the storage
#       target (including any switches between HBA and storage target), storage controller queue time, storage controller
#       time, storage device latency, storage controller status processing time, transport time back to the host (again through
#       any given switches, HBA time, driver short path to process the returning io completion interrupt.  While the actual
#       'storage latency time' itself within a storage box is reasonable, the queueing time within the storage controller can be
#       significant when the controller is under load from this, as well as other, hosts.  From a kernel perspective the whole
#       combination of hardware latencies are seen as a monolithic storage latency that cannot be further subdivided.
#  --------------------------------------------------------------------------------------- 
#
# Compact State Form:
# ==================================================================================
# 
# |merge|
# Q---->M------------>D----------------------------------------------------------->C    Merge: add into an existing request
# Q------>G->I------->D----------------------------------------------------------->C    Add  : create a new io request
# |--insert--|--queue-|                                                                 Q2D  : total scheduler/host time
#                     |--- driver,adapter,transport,switch,storage latency time ---|    D2C  : total storage latency (outside of OS)
# |---- host time ----|--- hardware based time ------------------------------------|    Q2C  : await time in iostat output
# 
# NOTE: Not to scale.  Earlie M,D|G,I,D events are spread out to allow string label descriptions.
# 
# 
# Q---->M                                                                             Q2M, [1] 'merge' time, time to find existing request and add this I/0 to it
# Q------>G                                                                           Q2G,     'create' time to allocate request data structure for new I/O
#         G->I                                                                        G2I, [2] 'sort' time, how long it took sort algorithm to insert I/O request into queue
# Q------>G->I                                                                        Q2I, [3] 'insert' time, time spent by sched on staging I/O
# Q------>G->I------->D                                                               Q2D,     'host' time, time io request spends in scheduler, also kernel|sched time
#            I------->D                                                               I2D, [4] 'queue' time, time spent in sched queue awaiting dispatch
#                     D----------------------------------------------------------->C  D2C,     'storage' time, this is dominated by storage latency time
# Q------------------------------------------------------------------------------->C  Q2C,     'await' time, time as reported via 'await' by iostat et.al.
# 
# 
# |<-host side time-->|<-------------------- hardware side time ------------------>|                          
# Q------------------>D                                                               Q2D,      host side time
#                     D----------------------------------------------------------->C  D2C, [5]  hardware based time (driver/adapter/transport/switch/storage)
# Q------------------------------------------------------------------------------->C  Q2C,      await time as reported by iostat, sar, etc.
# |<------------------------ await time ------------------------------------------>|
#
#
# Q------------------------------------------------------------------------------->Q  Q2Q, time between adajacent bio being set into io scheduler
#                                                                          C------>Q  C2Q, idle time -- but only if no other io present (qdepth=0 post C event)
#
# NOTES
# [1] : A merged request is simply added to an existing request already inserted/present within the scheduler.  As
#       such, the 'M'erge (or 'F'ront merge) event is typically the last event specifically associated with this bio.
#       Any M2D time is then associated with the request the I/O was merged into.  M2D times can increase both if
#       additional I/O are merged into the request and due to sorting.
# [2] : This is the time used by whatever scheduler algorithm is in use to 'sort' a request into the scheduler queue.
#       This time will vary by which scheduler is in use.  The noop and none options have near zero 'sort' time. Things
#       Like deadline and cfq a little more time, but in general sort time is a well behaved sort delay regardless.
#       Typically, both because G events may not be present and due to sort time being short and well behaved, the
#       more general Q2I time is used as the 'sort' time event though it is made up of 2 or more subcomponents.
# [3] : Not all implementations output S|G events.  As such, the Q2I time is more typically associated with 'sort time'
#       or queue time.  This is the total amount of time spent by the kernel staging the I/O into the scheduler.  Any 
#       delay from I2D is more driven by scheduler policy (such as idle slice time for cfq) and/or congestion within 
#       storage (such as lun queue depth being full so as to delay dispatch).  In general, Q2I time is more important
#       than any of the sub-components therein.
# [4] : The I2D time includes any scheduler personality delays, but is nominally any significant delays here will be
#       dominated by device|target|driver lun queue full conditions.  That is, typically the delay witihn I2D time that
#       is associated with kernel code is short.  Long delays here are almost always associated with lun queue depth
#       issues. Scheduler personality delays that are common in I2D, for example, is the slice idle time within cfq which
#       injects a short delay to allow for improved merging.  Similar delays are not present in deadline, noop, etc.
# [5] : the hardware based time includes the driver path which is fairly short.  The driver's main job is to repackage
#       the kernel request data structure into a form that the HBA's firmware can accept and then passing that mapped
#       request information into the controller's firmware -- typically via a queue/mailbox function.  This driver 
#       based code path is both short in duration and nominally stable.  Therefore the driver side portion of the 
#       D2C time is considered fixed and a small fraction of total D2C time.  Internal controller functions, such as
#       san buffer credits, can delay the HBA hardware from delaying the transmission of the io request to storage,
#       but the kernel has no further insight into those types of hardware based issues within the HBA controller itself.
#       Such time is counted against the hardware side of things since there is no kernel nor driver code involved in
#       same.
#
# 
# Long State Form:
# ==================================================================================
# 
#               +----------+
#               |          | 
#               |         {R}
#               |          ^
#               v          |
# A->Q+>{S}->G->I--+---->D-+--------------------------------------------------------->C    
#     |            ^
#     v  +--->M----+
#     +--+--->F----+
# 
# Key: { }- optional or unsual events that nominally should not be present
#       A - remAp   ; lba block mapping               M - Merge   ; merge io onto end of existing request on q
#       Q - Queue   ; io arrival @sched               F - Front   ; merge io onto front of existing request on q
#       S - Sleep   ; await for request               D - Dispatch; send req to driver
#       G - Getreq  ; get request struct              R - Requeue ; io rejected, place back on queue
#       I - Insert  ; insert req into q               C - Complete; io done 
#       X - SplitIO ; split a request in two          B - Bounce  ; bounced data page to reserved lower memory req'd
#       -----------------------------------------------------------------------
#       P - Plug    ; plug/stop draining io to drv    U - Unplug  ; unplug drain
#       T - Timeout ; unplug drain due to timeout
#
#NOTES:
#{....} : are events that may not happen
#{A-->} : remAp event, for example map the LBA (sector) from partition relative number to disk absolute sector number.
#{-->S} : Sleep; wait for availability of request data structure, typically when nr_request limit is reached, 
#{-->R} : Requeue: driver rejects the io even though its below advertised lun queue depth, typically a resource issue 
#         of some type exists within the driver and/or HBA causing this rejection of the io back to the scheduler.
#         Requeued I/O are placed at front of the queue -- next up -- and futher dispatch to the driver is stalled
#         until this requeued io is accepted.  It can be common to see an occassional 'R'equeue happen and is 
#         something to watch but not be overly concerned about.  However, if there are a large number of 'R'equeues
#         happening or a large number on some io then the root cause within the hardware should be investigated. Often
#         lowering the lun queue depth can help reduce 'R'equeues from happening.  Reducing nr_requests can have a 
#         similar affect via indirect means by slowing the number of io available at any given time for the scheduler
#         to use to fill the underlying lun queue.
#
# NOTE:   option flags will control how time associated with Requeues is accumlated, that is into either Q2D or D2C time.
#         Either accumulation is sub-optimal which is why its recommended to allow a D2D time to be added to the per event output.
#        events    Q------------>D->R-I->D->R->I->D------------>C 
#          +--- option flag controlling where R,I,D sequence is accumulated into.
#          v
#        0=storage |---Q2D------>|--------D2C------------------>| this is normal output but will complain about requeues
#        1=host    |---Q2D----------------------->|---D2C------>| if host centric output mode selected, will still complain
#        show D2D  |---Q2D------>|---D2D--------->|---D2C------>| best, host view Q2D reflects host time, D2C reflects storage
#                                                                 and D2D time kept separate.
#
